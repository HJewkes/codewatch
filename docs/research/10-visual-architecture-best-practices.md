# Visual Architecture Representations — Survey for codewatch

**Date**: 2026-05-10
**Purpose**: Survey canonical software-architecture views and pick the ones that fit codewatch's already-extracted IR (tree-sitter AST, dep graph in SQLite with closure tables, Martin metrics, Louvain/Leiden communities, smells). Goal: scannable, interactive visuals that replace prose plans for two jobs — *understand existing project* and *describe a planned change*.

---

## 1. C4 Model (Simon Brown)

Hierarchical zoom: **System Context -> Container -> Component -> Code** ([c4model.com](https://c4model.com/)).

| Level | Question | Audience | Generatable from codewatch? |
|---|---|---|---|
| Context | "What's the system, who uses it, what does it touch?" | Mixed (business + tech) ([icepanel.io](https://icepanel.io/blog/2024-07-18-what-is-the-c4-model)) | No — needs human-supplied actors/external systems. |
| Container | "What deployable units exist?" | Architects, devs, technically literate PMs | Partial — package boundaries map to containers in a monorepo, but runtime processes/DBs aren't in the IR. |
| Component | "What are the building blocks inside a container?" | Architects, devs ([codesee.io](https://www.codesee.io/learning-center/c4-model)) | **Yes** — Louvain/Leiden communities + Martin metrics already produce this. |
| Code | "What's the class structure?" | Devs only; "rarely done in practice and can ideally be autogenerated" ([codesee.io](https://www.codesee.io/learning-center/c4-model)) | Yes from AST, but Brown discourages drawing it manually. |

**Strength**: hierarchy = same audience model can zoom. **Weakness**: doesn't show data flow or change hotspots; misuses are common ([workingsoftware.dev](https://www.workingsoftware.dev/misuses-and-mistakes-of-the-c4-model/)). For codewatch, **Component-level is the natural target** — the boundary detector already produces it.

## 2. UML Survivors (2026)

UML 2.5 still defines 14 diagram types ([uml-diagrams.org](https://www.uml-diagrams.org/uml-25-diagrams.html)), but practical use has narrowed:

- **Sequence** — alive; dominant for API/microservice flows and runtime scenarios ([archimetric.com](https://www.archimetric.com/blog/2025/12/08/why-uml-isnt-just-surviving-in-2025-its-thriving-thanks-to-ai-and-visual-paradigm/), [lucidchart.com](https://www.lucidchart.com/pages/uml-sequence-diagram)).
- **Class** — alive for onboarding/domain model; over-engineered when used as "the design".
- **Component** — largely subsumed by C4 Container/Component.
- **Deployment** — alive for cloud topology; map services -> infra ([geeksforgeeks](https://www.geeksforgeeks.org/system-design/deployment-diagram-unified-modeling-languageuml/)).
- **Activity** — niche; flowchart-equivalent, often beaten by BPMN or sequence.
- **Use case, object, package, profile, composite structure, timing, communication, interaction overview** — mostly dead in agile/SaaS shops ([conceptviz.app](https://conceptviz.app/blog/uml-diagram-types-complete-guide)).

For codewatch: only **sequence** and **deployment** carry weight, and only sequence is partially auto-generatable.

## 3. Dependency / Module Graphs

A directed graph of files, modules, or packages — codewatch's native output. Layout choice dominates legibility:

- **Sugiyama / layered** — vertices in horizontal layers, edges directed downward; the canonical layout for "this depends on that" semantics. Used for flowcharts, UML, call graphs, ER diagrams ([Wikipedia: layered graph drawing](https://en.wikipedia.org/wiki/Layered_graph_drawing), [yWorks](https://www.yworks.com/pages/layered-graph-layout)).
- **ELK Layered** — the open-source production-grade Sugiyama; five-phase pipeline (cycle removal, layering, crossing minimization, node placement, edge routing) ([eclipse-elk DeepWiki](https://deepwiki.com/eclipse-elk/elk/2.1-layered-layout-algorithm)).
- **Force-directed** — attractive/repulsive forces minimizing energy; good for clustering exploration, bad when direction matters ([ona-book.org](https://ona-book.org/viz-graphs.html)).
- **d3-dag** — JS library specifically for DAG layouts ([github.com/erikbrinkman/d3-dag](https://github.com/erikbrinkman/d3-dag)).

Heuristic: **dep graphs almost always want hierarchical/Sugiyama** (direction is the whole point); use force-directed only as initial placement or for community-detection views without arrows.

**Weakness**: at >300 nodes, every layout becomes a hairball — must filter by community, depth, or smell.

## 4. Sequence vs Call Graph

Two views of dynamic behavior; people confuse them ([ilograph.com](https://www.ilograph.com/blog/posts/the-two-fundamental-types-of-architecture-diagrams/)).

- **Sequence**: lifelines + time axis + messages between participants. Best for cross-component runtime scenarios, API flows, "what happens when a user clicks X" ([Wikipedia: sequence diagram](https://en.wikipedia.org/wiki/Sequence_diagram), [softwaresystemdesign.com](https://softwaresystemdesign.com/software-architecture-design/modeling-and-documentation/sequence-diagrams/)).
- **Call graph**: nodes = functions, edges = calls; static, hierarchical, no time. Best for "what calls this", refactor blast-radius, dead-code detection.

Rule of thumb: **sequence when you care about ordering and external participants; call graph when you care about reachability and structure.** codewatch's IR yields call graphs cheaply; sequence requires either runtime tracing or scenario-driven hand-authoring.

## 5. Data-Flow / Pipeline Diagrams

Distinct from control flow: **DFDs have no decisions, no loops** — only how data moves through processes/stores ([Wikipedia: data-flow diagram](https://en.wikipedia.org/wiki/Data-flow_diagram), [rudderstack.com](https://www.rudderstack.com/blog/data-flow-diagram/)). For ETL/indexing/ingestion code (which codewatch itself is), this is often the *right* abstraction — boxes are stages, arrows are typed records. Modern variants (ETL graphs) add control-flow and dependency edges as separate edge types ([puppygraph.com](https://www.puppygraph.com/blog/etl-graph)).

Hand-drawn today; not derivable from a pure dep graph because edges in a DFD are records, not function calls. Would need annotation on pipeline-stage modules.

## 6. arc42

Documentation template (12 sections); the visual sections are ([arc42.org/overview](https://arc42.org/overview)):

- **§5 Building Block View** — static decomposition, hierarchical ([docs.arc42.org/section-5](https://docs.arc42.org/section-5/)). Equivalent to C4 Container+Component.
- **§6 Runtime View** — important runtime scenarios ([docs.arc42.org/section-6](https://docs.arc42.org/section-6/)). Sequence diagrams in practice.
- **§7 Deployment View** — infra mapping ([docs.arc42.org/section-7](https://docs.arc42.org/section-7/)).

arc42 is prose-heavy and template-rigid; it's a *table of contents* more than a visual language. Useful as a checklist of *which views to produce*, not as a notation.

## 7. High-Signal Niche Views

- **Hexagonal / Ports-and-Adapters / Onion** — concentric rings, dependencies point inward ([Wikipedia: hexagonal architecture](https://en.wikipedia.org/wiki/Hexagonal_architecture_(software)), [happycoders.eu](https://www.happycoders.eu/software-craftsmanship/hexagonal-architecture/)). Excellent for showing whether a planned change respects the dependency rule. Generatable if you tag domain vs adapter modules.
- **Deployment topology** — services + infra; not in codewatch's IR.
- **Request lifecycle** — sequence-flavored; manual.
- **Entity state machines** — model an object's lifecycle; one-per-entity, hand-drawn ([techtarget.com](https://www.techtarget.com/searchapparchitecture/definition/state-diagram-state-machine-diagram-or-statechart-diagram)). High signal when entities have non-trivial states.
- **Treemaps** — nested rectangles; size = LoC, color = complexity ([ndepend.com](https://www.ndepend.com/docs/treemap-visualization-of-code-metrics), [erik.doernenburg.com](https://erik.doernenburg.com/2010/05/metrics-tree-maps/)). Spots oversized/over-complex modules at a glance.
- **Hotspot map (CodeScene)** — hierarchical circle-pack, size = complexity proxy (LoC), color/intensity = change frequency; the overlap is the hotspot ([CodeScene docs](https://codescene.io/docs/guides/technical/hotspots.html)). The single highest-signal view in the literature for *finding debt*. Requires git log integration on top of codewatch's IR.

## Recommendation

Pick five views; the first three are auto-generatable from codewatch today, the last two need small additions.

1. **Component / Module dependency graph (Sugiyama-layered, community-colored)** — primary "understand the system" view. Nodes are modules, edges are imports, colors are Louvain communities, badges are Martin-metric outliers. Filter by depth and by community to defeat hairballs. **No new extraction needed.**

2. **Hotspot treemap (LoC x change-frequency)** — primary "where's the debt" view. Hierarchical circle-pack or treemap, size = LoC, color = change-frequency intensity, halo = smell density. Forces the question "is this plan addressing a hotspot or a dead corner?" **Needs git-log churn extractor on top of the existing IR** — a one-table addition.

3. **Planned-change overlay (diff-graph)** — for "describe a change", render the same dep graph with **add/remove/move** annotations on nodes and edges, plus delta values for Martin metrics (Ca, Ce, I, A, D). Lets a reviewer scan for redundancy ("this proposes a new module that already exists in community X") and excess coupling ("this adds 7 inbound edges to a leaf"). **Needs a planned-state IR** — same schema, marked as proposed; diffable against current.

4. **Boundary / hexagonal lens** — re-render the same module graph with concentric rings (domain core / application / adapters / infra). Catches dependency-rule violations instantly. **Needs module tagging** (`role: domain | adapter | infra`) — either heuristic from path, or annotation. Two- to three-day add.

5. **Runtime sequence diagram (hand-authored, scoped to one scenario per plan)** — the only genuinely manual view, intentionally. Plans should include exactly one scenario sequence ("user runs `codewatch index`") to anchor the change in observable behavior. Codewatch can't auto-generate this without runtime tracing, so keep it as a Mermaid block authored by the planner; codewatch validates the participant list against the dep graph (every lifeline must resolve to a real module).

**Skip**: full C4 Context (needs hand-drawn external actors, low ROI), UML class/deployment/activity (over-engineered or out-of-scope), pure call graph at function granularity (too dense to scan), full arc42 (a checklist, not a view). C4 *vocabulary* is fine to borrow — it's how reviewers will name the levels.

The wins: (1)+(2) answer "what's already there?" in two scannable views; (3)+(4) answer "does this plan respect the existing structure?" before anyone reads prose; (5) keeps a scenario in the loop. Together they expose redundancy, dead code, and excess links — the stated pain point — without asking the planner to draw anything they can't justify against extracted data.
